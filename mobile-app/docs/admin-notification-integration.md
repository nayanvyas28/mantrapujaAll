# Complete Architecture Guide: Mantra Puja Notification System

This document is the ultimate technical guide for connecting the external **Admin Panel Website** with the **Mantra Puja React Native App**. 

It details the end-to-end data flow, database schemas, API specifications, and the exact mechanisms of the In-App Inbox so that push notifications work robustly, seamlessly, and permanently.

---

## 1. High-Level Architecture Flow
When an administrator clicks "Publish" on a notification from the external website, the following sequence occurs:

1. **Admin Panel** sends an authenticated `POST` request with the notification payload to the **Mantra Puja Node.js Backend**.
2. **Node Backend** authenticates the request using an Admin Secret Key.
3. **Database Insertion:** The backend saves the notification into the Supabase `notifications` table so it persists forever in the users' In-App Inbox.
4. **Token Retrieval:** The backend fetches all valid `expo_push_token` entries from the Supabase `profiles` table.
5. **Expo Push Dispatch:** The backend chunks the payload into groups of 100 and sends them to the Expo Push API (`https://exp.host/--/api/v2/push/send`).
6. **Physical Alert:** Phones receive the Push Notification (sounding a chime and displaying a banner).
7. **In-App Inbox Sync:** When the user opens the app and navigates to the "Notifications" screen, it pulls the latest messages from the Supabase `notifications` table and cross-references them with `user_notification_reads` to check if they have unread messages.

---

## 2. Database Schema (Supabase)

To support both targeted and broadcast (global) notifications without duplicating data for 100,000+ users, we use a global broadcast schema paired with a user-specific read-receipt table.

### Table: `public.notifications`
Stores the actual notification payloads.
- `id` (UUID, Primary Key) - `gen_random_uuid()`
- `title` (TEXT, Not Null) - e.g. "Maha Shivratri Special!"
- `message` (TEXT, Not Null) - The body copy.
- `type` (VARCHAR, Default 'GENERAL') - Used on the frontend to determine icons ('PROMO', 'BOOKING_UPDATE', 'GENERAL').
- `image_url` (TEXT, Nullable) - HTTP URL for rich-media image banners.
- `target_user_id` (UUID, Nullable, FK to `profiles.id`) - If `NULL`, this message is broadcast to **everyone**. If a specific UUID is provided, it only syncs to that specific user's app inbox.
- `created_at` (TIMESTAMPTZ, Default `now()`)

### Table: `public.user_notification_reads`
Since a row in `notifications` might be broadcast to 10,000 people, we do not want to duplicate it. Instead, when a user views their inbox and taps a notification, we insert a read receipt here.
- `user_id` (UUID, FK to `profiles.id`)
- `notification_id` (UUID, FK to `notifications.id`)
- `read_at` (TIMESTAMPTZ, Default `now()`)
*Primary Key:* composite of `(user_id, notification_id)`

### Table: `public.profiles` (Update)
- added `expo_push_token` (TEXT, Nullable) - The unique APNs/FCM device token generated by Expo native code required to send a phone a push payload.

## 3. Backend API Specification (Node.js)

Your website developer must trigger this exact API endpoint whenever they wish to publish a notification.

**Endpoint:** `POST /api/admin/notifications/broadcast`
**Host:** Whatever URL hosts the Node.js backend (currently `http://localhost:4000` for testing).
**Headers:** 
- `Content-Type: application/json`

**Payload Model:**
```json
{
  "secret": "mantrapuja-admin-keys",
  "title": "Maha Shivratri Pooja",
  "message": "20% off all Poojas, book your slot now!",
  "type": "PROMO",
  "image_url": "https://example.com/shivratri-banner.jpg",
  "target_user_id": null
}
```
*Note: `target_user_id` and `image_url` are fully optional. If `target_user_id` is excluded, it broadcasts to all users.*

**Success Response (200 OK):**
```json
{
  "message": "Push Notifications dispatched successfully!",
  "db_record": {
    "id": "abc-123",
    "title": "Maha Shivratri Pooja",
    "message": "20% off all Poojas, book your slot now!",
    "type": "PROMO",
    "image_url": "https://example.com/shivratri-banner.jpg",
    "target_user_id": null,
    "created_at": "2026-03-02..."
  },
  "devicesAlerted": 50
}
```

**Backend Logic Handling:**
1. Validates the `secret` against `process.env.ADMIN_SECRET`.
2. Supabase Insert: `INSERT INTO notifications (title, message, type, image_url, target_user_id)`
3. Query `profiles` where `expo_push_token IS NOT NULL`.
4. Create the Expo Payload object for each token.
5. Loop chunks of 100 via Axios to `https://exp.host/--/api/v2/push/send`.

## 4. Frontend Implementation & App UI

### Part A: Registering the App Device Token (`hooks/usePushNotifications.ts`)
When the app launches (`app/_layout.tsx`), the custom `usePushNotifications()` hook is invoked.
1. It requests physical device permissions via `Notifications.requestPermissionsAsync()`.
2. It generates a unique `ExpoPushToken[xxxx-xxxx-xxxx]` tied to the device.
3. It fetches the active Supabase Session.
4. It secretly updates the user's `profiles` row in Supabase, inserting the newly generated `expo_push_token`.

### Part B: The In-App Inbox (`app/profile/notifications.tsx`)
The user's Notification screen is a fully functional email-style inbox that guarantees no notifications are missed even if they ignored the physical phone buzz.

**Inbox Data Flow:**
1. **Load:** The screen fires `supabase.from('notifications')` and downloads the latest 50 notifications. Because of Row Level Security (RLS), it ONLY downloads notifications where `target_user_id IS NULL` (Global Broadcasts) OR `target_user_id = user.id` (Personal).
2. **Read Receipts:** In parallel, it fetches `supabase.from('user_notification_reads')` for the active user.
3. **Merge:** The React Native state merges these arrays locally. If a `notification.id` is NOT found in the `read receipts` array, a visually distinct **Red Saffron Dot** is forced onto the UI card indicating an unread message.
4. **Interaction (Mark as Read):** When a user taps a notification card:
   - The UI optimistically hides the Red Saffron Dot.
   - An asynchronous query inserts a row into `user_notification_reads`: `INSERT INTO user_notification_reads (user_id, notification_id)`.

## 5. Security and Maintenance Best Practices

- **Row Level Security (RLS):** Supabase RLS is strictly configured on `user_notification_reads` so users can exclusively insert or view their own read receipts, and they can only view `notifications` deemed publicly broadcast or directed at their UUID.
- **Expo Push Token Refresh:** Tokens can expire. Because our `usePushNotifications` hook mounts on every app launch, it guarantees the database is silently hot-swapped with the freshest valid token automatically every time they use the app.
- **Admin Secret Vaulting:** The Node.js Webhook relies on an environment variable `ADMIN_SECRET`. In a production environment, this must be an extremely strong randomized cryptographic string.
- **Error Silence on Push:** If a user uninstalls the app but their token remains in `profiles`, the Expo Push API will return a `DeviceNotRegistered` error ticket. The Node.js backend ignores this to prevent crash-looping during mass broadcasts.

With this architecture, the admin panel operates independently of the app. A simple API payload is enough to natively buzz 10,000 phones and simultaneously deliver persistent, rich-media alerts safely into their private app inbox.
